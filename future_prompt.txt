I have an existing Laravel application that currently follows the default Laravel structure where controllers, models, policies, migrations, and other classes are mixed inside global directories such as `app/Models`, `app/Http/Controllers`, etc.

I need to refactor this application into an **enterprise-grade modular architecture** inspired by **Domain-Driven Design (DDD)** while keeping the system simple and synchronous (no queues or background jobs).

The goal is to make the system **scalable, maintainable, and optimized for financial reporting**, while supporting complex Excel-based financial data imports.

The current functionality must be grouped into a module called **Jamgarma**, which represents a financial development fund. In the future, another module called **Gazna** will be added, so the architecture must ensure strong separation between modules.

The system must also process **complex financial Excel reports** containing multi-level column headers and hierarchical financial categories such as income, expenses, reserves, and balances. These reports contain district-level financial data and may evolve over time with new financial categories.

The architecture must therefore support **dynamic financial categories** and avoid rigid database schemas that directly mirror Excel columns.
